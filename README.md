# main-project
Основной репозиторий проекта для практики

## Project Board
(https://github.com/orgs/practica-team/projects/1)


## Ветки

 - **main**  # Стабильная релизная ветка. Содержит только проверенный, протестированный код, готовый к развёртыванию. В неё запрещено коммитить напрямую —
изменения попадают только через Pull Request из develop. Работает на ней Тимлид / DevOps.
 - **develop** # Интеграционная ветка для тестирования. Сюда сливаются готовые фичи из *других рабочих веток*. Здесь проходят тесты и общее ревью. После
успешной проверки изменения переносятся в main. Работают *все разработчики (через PR*).
 - **backend** # Ветка бэкенд-разработчиков. Вся серверная часть (API, базы данных, бизнес-логика) разрабатывается здесь. После завершения фичи —
PR в develop. Работает б*экенд-команда*.
 - **docker** # Ветка DevOps-инженеров. Содержит Dockerfile, docker-compose.yml, скрипты развертывания, настройки CI/CD (.github/workflows). Изменения
инфраструктуры синхронизируются с develop, чтобы все разработчики имели актуальное окружение. Работает *DevOps-команда*.
 - **docs** # Ветка Аналитиков. Содержит техническую документацию, схемы БД, API-спецификации (OpenAPI/Rest API), руководства пользователя. Изменения
 вливаются в develop и затем в main, чтобы документация всегда была актуальна релизной версии. Работают *Аналитики*.
 - **frontend** # Ветка фронтенд-разработчиков. Весь клиентский интерфейс, верстка, взаимодействие с API разрабатывается здесь. После завершения фичи — PR
в develop. Рабатает *Фронтенд-команда*.
 - **qa**  # Ветка подготовки релиза (Quality Assurance). Создается от develop перед выкаткой. В этой ветке QA-инженеры прогоняют полный загон,
нагрузочные тесты и исправляют критичные баги. После успешного утверждения QA ветка вливается в main. Работают оманда *QA/Тестировщики*.

## Процесс работы

1. Берёте задачи из Projects и не забываете отмечать(все подродбности обсуждаете с командой)

<br>
<br>

2. Начало спринта
Каждый разработчик перед началом работы обязан обновить свою ролевую ветку из develop, чтобы взять последние изменения коллег:<br>
Например, для бэкендера:
    ```bash
    git checkout backend
    git pull origin backend
    git merge develop
    ```

Для DevOps (docker):
    ```bash
    git checkout docker
    git pull origin docker
    git merge develop
    ```
<br>
<br>

3. Разработка и коммиты.<br>
Все изменения делаются внутри ролевых веток. Коммиты создают по *определнному формату*.

<br>
<br>

4. Работа в командных ветках.<br>
Все изменения вливаете в ролевые ветки. Начальное етстирование

<br>
<br>

5. Пулл-реквест в develop.<br>
Когда задача выполнена и хорошо работает, создается Pull Request из ролевой ветки в develop.

Из: backend / frontend / docker / docs

В: develop

В PR обязательно:
* Описание изменений.
* Ссылка на задачу (Jira/Trello/GitHub Issue).
* Запрос ревью у коллег.
После одобрения и успешного прохождения CI (сборка, линтеры, юнит-тесты) — PR вливается в develop.

<br>
<br>

6. Релиз в main<br>
Когда QA подтверждает, что ветка qa стабильна:<br>
Создается Pull Request из qa -> main.<br>
После ревью и финальной проверки DevOps'ом PR вливается.<br>
Тег версии (например, v1.0.0) ставится на коммите в main для отслеживания релизов.<br>

<br>
<br>

7. Обратная синхронизация (важно!)
После релиза в main и не только, чтобы все ролевые ветки не отставали:

    ```bash
        git checkout main
        git pull origin main
        git checkout develop
        git merge main          
        git push origin develop
    ```

## Комммиты

Правилы создания коммитов:

### Формат коммита

 - тип(Тикет/isuue): описание

### Типы коммитов

 - 1. **feat** — Новая фича / функциональность
 - 2. **work** — Обычная работа/ написание кода
 - 3. **fix** — Исправление бага
 - 4. **docs** — Изменения в документации
 - 5. **style** — Форматирование, пробелы, точки с запятой
 - 6. **refactor** — Рефакторинг кода без изменения поведения
 - 7. **test** — Добавление/изменение тестов
 - 8. **chore** — Служебные изменения (зависимости, конфиги)

### Пример

- git commiit -m "work(DEV-32): изменил цвет у компонета React

## Формат написания кода

* Максимальная длинна стороки кода 140 символов<br>
* Клавиша Tab равна 4 пробелам<br>
* Разрешенно использование двойных и одинарных кавычек<br>
